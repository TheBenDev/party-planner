# packages/proto — CLAUDE.md

Protobuf service definitions for the party-planner API. Source of truth for all RPC contracts between `apps/web` and `apps/api`. See root CLAUDE.md for monorepo conventions.

## Tooling

- **buf CLI** for linting + code generation
- `buf.yaml` — module config + lint rules
- `buf.gen.yaml` — generation targets: Go stubs → `apps/api/gen/`, TypeScript stubs → `apps/web/src/gen/`

**After any `.proto` change:**
```bash
cd packages/proto && buf generate
```
Both apps must be rebuilt after generation.

## Adding a New Service

1. Create `planner/v1/<entity>.proto`
2. Define the service, messages, and enums following existing patterns
3. Run `buf generate` from this directory
4. Implement Go handler in `apps/api/internal/rpc/<entity>.go`
5. Register in `apps/api/cmd/api/main.go`
6. Add ConnectRPC client in `apps/web/src/lib/api/transport.ts`
7. Add oRPC router in `apps/web/src/server/routers/<entity>.ts`

## Key Message Notes

### SessionService (`session.proto`)
- `Session` message includes `poll_id` (optional string) and `announced_at`
- `Poll` and `PollAnswer` messages defined for Discord poll integration
- RPCs: `AnnounceSession`, `PollSession`, `GetSessionPoll`

### MemberService (`member.proto`)
- `InvitationStatus` enum: PENDING / ACCEPTED / EXPIRED / DECLINED / REVOKED
- `GetCampaignInvitationByTokenResponse` includes `sent_by` and `campaign_title`

### NonPlayerCharacterService (`non_player_character.proto`)
- `CharacterStatus` enum: UNKNOWN / ALIVE / DEAD / MISSING / SUSPICIOUS
- `RelationToParty` enum: uses `ENEMY` (not `HOSTILE` — see enum mismatch below)
- Foundry VTT fields on `Npc` message

### CampaignIntegrationService (`campaign_integration.proto`)
- Only `DISCORD` source in `IntegrationSource` enum; FOUNDRY is a future addition
- `metadata` and `settings` are `google.protobuf.Struct` (flexible JSON)

### QuestService (`quest.proto`)
- `reward` field is `google.protobuf.Struct` with `// TODO: fill out schema for reward`

### DiscordService (`discord.proto`)
- Defines availability RPCs (SetAvailability, GetAvailabilities, RemoveAvailability, ClearAvailability) and session/NPC/messaging RPCs
- **Not implemented as a ConnectRPC service in the Go API**
- This proto is the target contract for the discord.ts rework in `apps/web`

## Known Enum Mismatch

`RelationToParty` is inconsistent across layers:

| Layer | Value |
|---|---|
| `packages/proto` | `RELATION_TO_PARTY_ENEMY` |
| `apps/api/internal/models` | `"ENEMY"` |
| `packages/enums/src/character.ts` | `"HOSTILE"` |

Data written through Go stores `"ENEMY"`. Data written through web app Drizzle (character.ts violation) stores `"HOSTILE"`. Must be reconciled before character features are complete.

## Hard Rules

- Do not edit files in `apps/api/gen/` or `apps/web/src/gen/` — they are regenerated by `buf generate`
- Keep `buf generate` output committed; never gitignore generated stubs
- All new RPCs go through a proto service definition first — no ad-hoc HTTP endpoints for new features
- Field naming: use `snake_case` in proto; generated TS uses `camelCase` automatically
